## Foundation The solution follows the [Traditional "N-Layer" architecture applications](https://learn.microsoft.com/en-us/dotnet/architecture/modern-web-apps-azure/common-web-application-architectures#traditional-n-layer-architecture-applications) ‚Äî because some things never go out of style! In this architecture, users make requests through the API layer, which interacts only with the Domain layer. The Domain layer, in turn, can call the Model layer for data access requests. The API layer shouldn't make any requests to the Model layer directly, nor should it interact with persistence directly through other means. Likewise, the Domain layer should only interact with persistence by going through the Model layer. This ensures that each layer maintains its own well-defined responsibility. This design improves reusability. Keeping the Domain project independent allows me to reuse it in non-API projects. For example, a Service Bus message consumer or a background cron-based service can utilize the same business logic as the API, ensuring consistency while enabling different ways of triggering domain actions beyond HTTP requests. Another key advantage of this separation is improved testability. Structuring the code this way allows me to write tests in a more efficient and isolated manner. Once the Domain project is thoroughly tested, I can focus on testing the API layer without needing to retest business logic. This means I only need to write tests for API-specific behavior, such as request handling, authentication, and response formatting. In theory, as long as the API layer follows the contract defined by the Domain layer, changes at the API level won't break lower layers‚Äîthough the opposite is definitely not true. ## Step 1: Initializing the Solution and Projects To start, we'll create the solution and projects using the following structure: ```text Acme ‚îú‚îÄ‚îÄ src ‚îÇ ‚îú‚îÄ‚îÄ Acme.Core ‚îÇ ‚îú‚îÄ‚îÄ Acme.Model ‚îÇ ‚îú‚îÄ‚îÄ Acme.Domain ‚îÇ ‚îú‚îÄ‚îÄ Acme.Api.REST ‚îÇ ‚îú‚îÄ‚îÄ Acme.Api.GraphQL ‚îÇ ‚îî‚îÄ‚îÄ Acme.Auth ‚îî‚îÄ‚îÄ test ‚îú‚îÄ‚îÄ Acme.Testing ‚îú‚îÄ‚îÄ Acme.Core.Tests ‚îú‚îÄ‚îÄ Acme.Model.Tests ‚îú‚îÄ‚îÄ Acme.Domain.Tests ‚îú‚îÄ‚îÄ Acme.Api.REST.Tests ‚îú‚îÄ‚îÄ Acme.Api.GraphQL.Tests ‚îî‚îÄ‚îÄ Acme.Auth.Tests ``` - **Acme.Core** will act like a framework, containing code that can be reused by all other projects. The same will be the Acme.Testing, but for testing projects only. - **Acme.Model** will hold entities, models, and persistence infrastructure (e.g. DbContext, Repositories, UnitOfWork). - **Acme.Domain** will hold the business logic. - **Acme.Api.REST** and Acme.Api.GraphQL will have almost the same features and I will use them to showcase the differences between REST and GraphQL. - **Acme.Auth** will hold the authentication (and authorization). Initially, this will also be a REST API, but I‚Äôm exploring alternatives like AWS Cognito and Keycloak, so this might change. Before I begin, to ensure your code stays properly formatted when making changes and saving, please copy the following .editorconfig file to the root directory of your project. If you're using Visual Studio, I also recommend checking out this article first: [Must-Have Visual Studio Extensions for 2025](must-have-visual-studio-extensions.html) for an optimal development setup. ### Script for Initialization Here are the commands that automate the initialization process: ```BASH #!/bin/bash # Create an empty Solution file dotnet new sln -n Acme # Create and add core projects dotnet new classlib -o src/Acme.Core dotnet sln add src/Acme.Core dotnet new classlib -o src/Acme.Model dotnet sln add src/Acme.Model dotnet new classlib -o src/Acme.Domain dotnet sln add src/Acme.Domain dotnet new webapi -o src/Acme.Api.REST dotnet sln add src/Acme.Api.REST dotnet new webapi -o src/Acme.Api.GraphQL dotnet sln add src/Acme.Api.GraphQL dotnet new webapi -o src/Acme.Auth dotnet sln add src/Acme.Auth # Create and add test projects dotnet new xunit -o test/Acme.Testing dotnet sln add test/Acme.Testing dotnet new xunit -o test/Acme.Core.Tests dotnet sln add test/Acme.Core.Tests dotnet new xunit -o test/Acme.Model.Tests dotnet sln add test/Acme.Model.Tests dotnet new xunit -o test/Acme.Domain.Tests dotnet sln add test/Acme.Domain.Tests dotnet new xunit -o test/Acme.Api.REST.Tests dotnet sln add test/Acme.Api.REST.Tests dotnet new xunit -o test/Acme.Api.GraphQL.Tests dotnet sln add test/Acme.Api.GraphQL.Tests dotnet new xunit -o test/Acme.Auth.Tests dotnet sln add test/Acme.Auth.Tests # Check the build and test dotnet build -v q dotnet test -v q ``` This script initializes the solution and adds all necessary projects, ensuring they're correctly included in the solution. ## Step 2: Configuring Project Dependencies Next, set up references between projects to ensure proper communication and data flow. Here's how: - Adding reference to Acme.Core ```bash #!/bin/bash # main projects dotnet add src/Acme.Model reference src/Acme.Core dotnet add src/Acme.Domain reference src/Acme.Core dotnet add src/Acme.Api.REST reference src/Acme.Core dotnet add src/Acme.Api.GraphQL reference src/Acme.Core dotnet add src/Acme.Auth reference src/Acme.Core # test projects dotnet add test/Acme.Core.Tests reference src/Acme.Core ``` - Adding reference to Acme.Model ```bash #!/bin/bash # main projects dotnet add src/Acme.Domain reference src/Acme.Model # test projects dotnet add test/Acme.Model.Tests reference src/Acme.Model ``` - Adding reference to Acme.Domain ```bash #!/bin/bash # main projects dotnet add src/Acme.Api.GraphQL reference src/Acme.Domain dotnet add src/Acme.Api.REST reference src/Acme.Domain dotnet add src/Acme.Auth reference src/Acme.Domain # test projects dotnet add test/Acme.Domain.Tests reference src/Acme.Domain ``` - Adding reference to Acme.Api* projects As these projects are the top layer, only test projects needs to reference them. ```bash #!/bin/bash # test projects dotnet add test/Acme.Api.GraphQL.Tests reference src/Acme.Api.GraphQL dotnet add test/Acme.Api.REST.Tests reference src/Acme.Api.REST dotnet add test/Acme.Auth.Tests reference src/Acme.Auth ``` - Adding reference to Acme.Testing ```bash #!/bin/bash # test projects dotnet add test/Acme.Core.Tests reference test/Acme.Testing dotnet add test/Acme.Model.Tests reference test/Acme.Testing dotnet add test/Acme.Domain.Tests reference test/Acme.Testing dotnet add test/Acme.Api.GraphQL.Tests reference test/Acme.Testing dotnet add test/Acme.Api.REST.Tests reference test/Acme.Testing dotnet add test/Acme.Auth.Tests reference test/Acme.Testing ``` ## Enforcing Zero Warning Tolerance and Enabling XML documentation **Why do this?** Warnings can lead to unnoticed bugs, and XML documentation helps maintain clear communication between developers. Let's enforce these configurations. The documentation will later be used to generate API documentation automatically, streamlining development processes. The efforts spent on maintaining code quality now will translate into time savings during API documentation generation in the future (I hope you care about it). The following settings are added to each project's .csproj file: ```xml <PropertyGroup> <TreatWarningsAsErrors>true</TreatWarningsAsErrors> <GenerateDocumentationFile>true</GenerateDocumentationFile> </PropertyGroup> ``` After applying it, we need to remove the auto-generated empty classes, otherwise build will fail. And, of course, I have a script for this step: ```bash #!/bin/bash # Zero warning tolerance and documentation # Optional, but I highly recommend to keep # Define root folders srcFolder="src" testFolder="test" # Generate documentation file setting (to be added inside specific PropertyGroup) generateDocumentationContent=" <GenerateDocumentationFile>true</GenerateDocumentationFile>" # PropertyGroup settings to add to all .csproj files insertContent=' <PropertyGroup Condition="'\''$(Configuration)|$(Platform)'\''=='\''Release|AnyCPU'\''"> <TreatWarningsAsErrors>True</TreatWarningsAsErrors> </PropertyGroup> <PropertyGroup Condition="'\''$(Configuration)|$(Platform)'\''=='\''Debug|AnyCPU'\''"> <TreatWarningsAsErrors>True</TreatWarningsAsErrors> </PropertyGroup>' # Function to check if setting exists contains_setting() { local file_content="$1" local setting="$2" echo "$file_content" | grep -q -F "$setting" } # Process all .csproj files in src folder echo -e "${GREEN}Processing src project files...${RESET}" find "$srcFolder" -type f -name "*.csproj" | while read -r csprojPath; do content=$(cat "$csprojPath") # Add GenerateDocumentationFile only to PropertyGroup with <ImplicitUsings>enable</ImplicitUsings> if ! contains_setting "$content" "GenerateDocumentationFile"; then sed -i '/<ImplicitUsings>enable<\/ImplicitUsings>/a\'"$generateDocumentationContent" "$csprojPath" echo -e "${GREEN}Added GenerateDocumentationFile inside PropertyGroup with ImplicitUsings for: $csprojPath${RESET}" else echo -e "${RED}Skipped GenerateDocumentationFile (ImplicitUsings not found) in: $csprojPath${RESET}" fi # Check and add TreatWarningsAsErrors if ! contains_setting "$content" "TreatWarningsAsErrors"; then # Add PropertyGroup settings after any closing PropertyGroup tag printf "%s\n" "$insertContent" | sed -i "/<\/PropertyGroup>/r /dev/stdin" "$csprojPath" echo -e "${GREEN}Added TreatWarningsAsErrors to: $csprojPath${RESET}" else echo -e "${RED}Skipped TreatWarningsAsErrors in: $csprojPath${RESET}" fi done # Process all .csproj files in test folder (skip GenerateDocumentationFile) echo -e "${GREEN}Processing test project files...${RESET}" find "$testFolder" -type f -name "*.csproj" | while read -r csprojPath; do content=$(cat "$csprojPath") # Check and add TreatWarningsAsErrors if ! contains_setting "$content" "TreatWarningsAsErrors"; then printf "%s\n" "$insertContent" | sed -i "/<\/PropertyGroup>/r /dev/stdin" "$csprojPath" echo -e "${GREEN}Added TreatWarningsAsErrors to: $csprojPath${RESET}" else echo -e "${RED}Skipped TreatWarningsAsErrors in: $csprojPath${RESET}" fi done # Remove auto-generated empty classes, otherwise build will fail echo -e "${GREEN}Removing auto-generated files...${RESET}" find src -name "Class1.cs" -delete find test -name "UnitTest1.cs" -delete ``` ## Step 4: Adding the must-have .NET Nuget packages The analyzer NuGet packages help us detect problems at build time, and since the projects are configured to consider warnings as errors, they won't be ignored (as we developers usually do). - IDisposableAnalyzers and Microsoft.VisualStudio.Threading.Analyzers are focused on helping with resource management and threading safety in your code. - SonarAnalyzer.CSharp is a comprehensive static code analyzer to improve the quality and security of your codebase. - FluentAssertions.Analyzers, NSubstitute.Analyzers.CSharp and xunit.analyzers help improve the readability, effectiveness, and correctness of tests. The other Nuget packages in the Acme.Testing are personal preference. I'm adding them now as I'm sure I will use them later. - Bogus generates fake data for testing, like names, addresses, and dates. - NSubstitute simplifies mocking in unit tests. - FluentAssertions makes test assertions more readable and expressive. - As alternative, you could use AutoFixture, Moq and Shouldly. ### Remove duplicated dependencies on test projects All test projects will reference Acme.Testing, which will serve as the core library for test projects. This library will include the essential dependencies (xUnit, runner, Coverlet) by default. The only package that must still be maintained is Microsoft.NET.Test.Sdk, as it is necessary for the tests to run, as outlined in the log message: ```text The following test container was not found: '‚Ä¶\nzr.acme\test\Acme.Model.Tests\bin\Debug\net9.0\Acme.Model.Tests.dll'. This can be resolved by one or more of the following steps: 1. The test container does not exist on disk and the corresponding project might need to be built successfully. 2. For .NET Core based test projects, please ensure that a nuget package reference to "Microsoft.NET.Test.SDK" exists and that it uses the latest stable version. üëàüëàüëà 3. For .NET based test projects, the project might be marked as a non-test project through an msbuild property, "IsTestProject". Please consider clearing it or set "<IsTestProject>true</IsTestProject>" in a "<PropertyGroup>" in the test project. ``` So the following references could be removed from all test projects except `Acme.Testing`: ```xml <PackageReference Include="coverlet.collector" Version="6.0.2" /> <PackageReference Include="xunit" Version="2.9.2" /> <PackageReference Include="xunit.runner.visualstudio" Version="2.8.2" /> ``` And of course, I have a script: ```bash #!/bin/bash # List of projects projects=( "test/Acme.Core.Tests" "test/Acme.Model.Tests" "test/Acme.Domain.Tests" "test/Acme.Api.GraphQL.Tests" "test/Acme.Api.REST.Tests" "test/Acme.Auth.Tests" ) # List of packages to remove packages=( "coverlet.collector" "xunit.runner.visualstudio" "xunit" ) # Loop through each project for project in "${projects[@]}"; do echo "Removing packages from $project" # Loop through each package to remove for package in "${packages[@]}"; do echo " Removing $package from $project" dotnet remove "$project" package "$package" done done echo "Packages removed successfully from all projects." ``` ## Step 5: Cleaning up the code Now it is time to open Visual Studio and what is still left.
