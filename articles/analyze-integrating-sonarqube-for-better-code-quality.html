Lacking a proper code analysis tool, teams may struggle with hidden bugs, security risks, and inconsistent code quality, leading to longer review times, increased maintenance costs, and potential issues in production. In this article, I'll walk you through how I enhanced the Nzr.Acme solution by integrating GitHub Actions with SonarQube for automated static code analysis. This is the fourth article in the "DevOps Days" series for the Acme project. If you missed the previous one check the related content at the end of this page. --- ## Why SonarQube? SonarQube is a powerful platform for continuous code quality and security analysis. It scans your source code, identifies issues, and provides actionable feedback. Integrating SonarQube with your CI pipeline ensures that every commit and pull request is analyzed, keeping your codebase clean and secure. --- ## Setting Up SonarQube for Your GitHub Repository This task is divided into 5 main steps: 1. Create a SonarQube account and organization. 2. Create a SonarQube project for the GitHub repository. 3. Testing the Integration with SonarQube 4. Fine-tune the analysis scope. 5. Prepare the GitHub project to run the workflow. I've also added a troubleshooting section for common issues you might encounter. Let's dive in. ## Create a SonarQube Account and Organization - Head over to [SonarQube](https://www.sonarsource.com/products/sonarcloud/signup/) and sign up using your GitHub account for easy integration. - Once logged in, click **Import an organization**. - You'll be redirected to GitHub to confirm the installation of SonarQube. Choose the **Only select repositories** option, then select your repository. You can update this later if needed. ![Repository Selection](../assets/articles/analyze-integrating-sonarqube-for-better-code-quality/9ef2462f-4056-47bb-b0a8-881a4fe42f5c.png) - Click **Install** to complete the setup. You'll then be redirected back to SonarCloud. - On the **Create an organization** page, confirm the **Name** and **Key**. The key is a unique identifier needed for your analysis configuration. - Choose a plan. I selected the **Free plan**, which supports up to 50k lines of code for private repositories or unlimited for public ones. - Finally, click **Create Organization**. ![Create Organization](../assets/articles/analyze-integrating-sonarqube-for-better-code-quality/ca773518-7679-40e0-9934-ec9e51792a53.png) ## Create the SonarQube project for your GitHub repository - After creating the organization, go to **My Projects** and click [Analyze new project](https://sonarcloud.io/projects/create). - Select your repository from the list: ![Analyze Projects](../assets/articles/analyze-integrating-sonarqube-for-better-code-quality/b28890b6-cdb7-4b79-a169-2d6d0a698af2.png) - Next, in the **Set up project for Clean as You Code** section: - Choose **Previous version** to compare changes against the previous codebase.

For my project, I excluded files generated during testing, such as .snap files. This ensures that SonarQube focuses only on relevant source files. If I need to exclude or include additional file types in the future, I can modify these settings without altering the workflow configuration. ![Source File Exclusions](../assets/articles/analyze-integrating-sonarqube-for-better-code-quality/daf97c65-8541-45e5-b9fe-30e65d96f4c5.png) ## Prepare GitHub to run the workflow - Go back to the **Analysis Method** page (you can access this via the **‚öôÔ∏è Administration** menu) and select **With GitHub Actions**. - Navigate to your GitHub repository settings using the link provided in SonarQube (or access https://github.com/<your-user>/<user-repository>/settings/secrets/actions) - Click **New repository secret**. - In the Name field, enter SONAR_TOKEN. - In the Value field, enter your SonarQube token. - Repeat the process to store the following additional secrets. Note that I replaced **'/'** with '-' when defining parameters. --- ## Building the Workflow The GitHub Actions workflow for building and analyzing the solution integrates SonarCloud to run static code analysis. This workflow ensures that every change is evaluated for code quality and security vulnerabilities. Here's how it works: ### Head: Name and Triggers At the top of the script, we define the workflow's name and triggers: ```yaml name: Build & Analyze the Solution # This GitHub Actions workflow automates building and analyzing the solution with SonarCloud. # It integrates static code analysis into the CI pipeline, ensuring code quality and identifying potential issues. # The workflow runs on every push or pull request targeting the 'main' branch, providing actionable insights. on: # Triggers push: branches: - main pull_request: branches: - main jobs: build-and-analyze: runs-on: ubuntu-latest steps: ``` The head of the workflow defines its purpose and specifies the events that trigger it: - **Name**: The workflow is named Build & Analyze the Solution to clearly indicate its purpose. - **Triggers**: The workflow listens for two events: pushes to the main branch and pull requests targeting the main branch. This ensures that any changes to critical branches are verified before merging. ### Jobs: Build and Test ```yaml jobs: build-and-analyze: runs-on: ubuntu-latest ``` The workflow contains a single job, `build-and-test`, which runs on an `ubuntu-latest` virtual machine. This job is divided into six steps, each with a specific purpose. ### Step 1: Checkout Code ```yaml steps: # Step 1: Checkout the code with full history - name: Checkout code uses: actions/checkout@v3 with: fetch-depth: 1 ``` The `actions/checkout` action fetches the repository's code onto the runner. This step is crucial for making the codebase available for subsequent steps. ### Step 2: Set Up .NET ```yaml # Step 2: Set up .NET - name: Set up .NET uses: actions/setup-dotnet@v3 with: dotnet-version: '9.0' ``` The `actions/setup-dotnet` action sets up the required .NET runtime and SDK on the runner. Here, I've specified .NET version 9.0 to align with the project requirements. ### Step 3: Cache NuGet Packages ```yaml # Step 3: Cache NuGet packages - name: Cache NuGet packages uses: actions/cache@v3 with: path: ~/.nuget/packages key: ${{ runner.os }}-nuget-${{ hashFiles('**/*.csproj', '**/*.fsproj', '**/global.json') }} restore-keys: | ${{ runner.os }}-nuget- ``` Caching NuGet packages speeds up builds by storing dependencies. The cache key is generated based on the operating system and the project's configuration files. ### Step 4: Restore Dependencies ```yaml # Step 4: Restore dependencies - name: Restore dependencies run: dotnet restore ``` This step restores the dependencies required for the solution using the dotnet restore command. The `|| exit 1` is for fail fast approach: If any errors occur, the step fails immediately to avoid unnecessary builds. ### Step 5: Build Solution ```yaml # Step 5: Build the solution - name: Build solution run: dotnet build --no-restore --configuration Release ``` This first `dotnet build --no-restore --configuration Release` step before starting SonarCloud analysis is there to ensure that if the build fails, you don't waste any units on SonarCloud analysis. If the build is broken, the SonarCloud analysis wouldn't even be triggered, so it's a good safety measure to check the build first before committing resources to the analysis. ### Step 6: Install SonarScanner for .NET ```yaml # Step 6: Install SonarScanner for .NET - name: Install SonarScanner for .NET run: dotnet tool install --global dotnet-sonarscanner ``` The SonarScanner tool is installed globally on the runner. This tool is essential for performing the SonarCloud analysis. ```yaml # Step 7: Run SonarCloud analysis - name: Start SonarCloud analysis env: SONAR_ORGANIZATION_NAME: ${{ secrets.SONAR_ORGANIZATION_NAME }} SONAR_ORGANIZATION_KEY: ${{ secrets.SONAR_ORGANIZATION_KEY }} SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }} run: | dotnet sonarscanner begin \ -o:"${SONAR_ORGANIZATION_NAME}" \ -k:"${SONAR_ORGANIZATION_KEY}" \ -d:sonar.host.url="https://sonarcloud.io" \ -d:sonar.login="${SONAR_TOKEN}" - name: Build project for SonarCloud run: dotnet build --no-restore --configuration Release - name: Finalize SonarCloud analysis env: SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }} run: dotnet sonarscanner end -d:sonar.login="${SONAR_TOKEN}" ``` The SonarCloud analysis is divided into three steps: 1. **Start Analysis**: The dotnet sonarscanner begin command initializes the SonarCloud analysis, using the SONAR_TOKEN secret for authentication. 2. **Build Project**: The project is built again to collect data for analysis. 3. **Finalize Analysis**: The dotnet sonarscanner end command completes the analysis and sends the results to SonarCloud.
